# 产品设计与成本控制说明

本文档用于说明本项目的功能计划、最终实现情况，以及运营成本控制策略。更完整的用户故事、状态机和交互细节见 [design.md](design.md)。

## 功能计划与实现情况

项目最初目标是做一个手机 Safari 可演示的 AI 视觉对话助手：用户打开摄像头后，用语音询问当前画面，系统抓取当前视频帧并让多模态模型回答。

| 功能方向 | 计划实现 | 最终实现 | 状态 |
|----------|----------|----------|------|
| 手机端摄像头预览 | 打开页面后启用后置摄像头，作为主视觉区域 | 实现全屏摄像头预览、权限状态、失败重试和关闭摄像头 | 已实现 |
| 语音提问 | 用户通过麦克风说出问题 | 实现麦克风授权、Web Speech API 识别、最终语音文本采集 | 已实现 |
| 自动视觉问答链路 | 语音结束后自动抓取当前画面并提问 AI | 实现语音稳定后自动停止聆听、抓帧、提交多模态请求 | 已实现 |
| 多模态 AI 回答 | 图片帧 + 文本问题发送给视觉模型 | 实现本地后端代理调用 DashScope OpenAI-compatible `qwen-vl-plus` | 已实现 |
| 语音回答与字幕 | AI 回答后自动朗读并显示字幕 | 实现浏览器 TTS、字幕浮层、重播和停止朗读；补充首次播放预热 | 已实现 |
| Orb 状态反馈 | 用一个可视化实体表达 AI 状态 | 实现 idle / listening / thinking / speaking 四态 Orb 和思考遮罩 | 已实现 |
| 多轮上下文 | 支持连续追问 | 实现当前页面内最近 4 轮文本上下文，不持久化历史 | 已实现 |
| 截图提问兜底 | 不说话时也能问当前画面 | 实现点击画面抓帧并使用默认视觉问题 | 已实现 |
| 文本输入兜底 | 语音识别不可用时仍能提问 | 实现显式文本问题输入，复用原有抓帧与提问链路 | 已实现 |
| 弱网反馈 | 网络慢或 AI 请求失败时给出恢复路径 | 实现 8 秒慢响应提示、失败后保留问题和画面、重试按钮 | 已实现 |
| 抓帧质量选择 | 在速度和准确度之间切换 | 实现低清 / 高清两档抓帧模式，切换模式时清空旧帧 | 已实现 |
| 回答复用 | 用户能复制回答内容 | 实现回答摘要、复制摘要、复制全文和失败提示 | 已实现 |
| 真机测试文档 | 方便按 iOS Safari 完整验收 | 新增端到端真机测试清单，覆盖权限、ngrok、弱网和回归边界 | 已实现 |

## 未纳入 MVP 的功能

| 功能 | 未做原因 |
|------|----------|
| 登录系统 | 演示核心是视觉问答闭环，登录会增加用户进入成本 |
| 历史记录持久化 | 本项目不保存长期对话，避免隐私和存储复杂度 |
| 连续视频流理解 | 成本和带宽不可控，且 MVP 场景只需要关键帧问答 |
| 服务端语音合成 | 会增加音频生成成本；浏览器 TTS 足够支撑演示 |
| 多用户账号与云端会话 | 与 3 天 MVP 的验证目标无关 |

## 成本控制思路

项目设计时重点考虑两类成本：模型调用成本和用户等待成本。

| 成本问题 | 设计思路 | 实际采用 |
|----------|----------|----------|
| 视频流上传成本高 | 不连续上传视频，只在提问时抓取单帧 | 已采用 |
| 图片过大导致等待久 | 抓帧后压缩 JPEG，弱网优先低体积 | 已采用 |
| 不同场景对清晰度要求不同 | 提供低清 / 高清模式，让用户按场景选择 | 已采用 |
| 多轮对话重复上传图片会浪费 token | 历史上下文只保留文本，不重复上传历史图片 | 已采用 |
| 语音识别和语音合成可能产生额外服务成本 | 使用浏览器 Web Speech API 和 TTS | 已采用 |
| 前端暴露 API key 风险高 | 增加本地 Node 后端代理，密钥只在 `.env` 和 Node 进程中 | 已采用 |
| 弱网失败导致用户重复操作 | 失败时保留当前问题和画面，允许直接重试 | 已采用 |
| 每次回答都再调用 AI 做摘要会增加成本 | 摘要用浏览器端首句/截断规则生成 | 已采用 |

## 实际采用的关键策略

### 1. 单帧抓取而不是视频流

系统只在用户提问时抓取当前视频帧，并把这一帧和文本问题发送给模型。这样可以避免连续视频流带来的带宽消耗、模型 token 消耗和延迟风险。

### 2. 自适应 JPEG 压缩

低清模式优先将图片压到约 40KB，适合蜂窝网络或弱网演示；高清模式放宽到约 120KB，适合文字、小物体和复杂画面。压缩后的尺寸和体积会显示在页面中，便于测试和调参。

### 3. 文本上下文，不重复传图

多轮上下文只保留最近 4 轮文本问答。下一轮提问时带入文本上下文，但图片只使用当前抓取的画面，避免历史图片重复消耗带宽和模型输入成本。

### 4. 浏览器端语音能力

语音识别使用 Web Speech API，回答朗读使用浏览器 TTS。这样不需要额外部署语音识别或音频生成服务，也不会产生独立音频模型调用成本。

### 5. 本地后端代理保护密钥

前端只请求本地 `/api/vision-chat`。真实 `AI_API_KEY` 由 `backend/server.mjs` 从 `.env` 读取，避免密钥进入前端构建产物、浏览器 JS 或 localStorage。

### 6. 失败恢复减少重复请求

AI 请求失败后，页面明确提示“问题和画面已保留”，用户可以直接重试，不需要重新说话或重新抓帧。这降低了失败后的重复操作和重复采集成本。

## 验证方式

- 本地构建：`npm run build`
- 后端健康检查：`GET http://127.0.0.1:8787/api/health`
- 桌面页面检查：`http://127.0.0.1:5173/`
- 手机真机检查：通过 ngrok HTTPS 地址在 iOS Safari 打开
- 完整真机用例见 [e2e-device-test-checklist.md](e2e-device-test-checklist.md)
