# AI 视觉对话助手设计文档

## 设计目标

这个项目解决的问题不是“把摄像头画面传给 AI”，而是让手机用户用最短路径完成一次视觉问答：打开页面、授权、提问、看到和听到回答。

MVP 的核心目标：

- 手机 Safari 可演示。
- 用户不需要理解模型、帧率、上下文、音频管线等技术细节。
- 系统按需抓取单帧，避免连续视频流上传带来的带宽和模型成本。
- AI 状态必须可感知，用户能区分系统正在听、正在思考、正在说话还是等待输入。

## 目标用户

### 用户 1：临时需要视觉辅助的手机用户

场景：用户想快速询问眼前画面，例如桌面上的物品、屏幕内容、菜单或提示牌。

用户故事：

- 作为用户，我希望打开页面后直接看到摄像头画面，而不是先进入说明页。
- 作为用户，我希望说出问题后系统自动使用当前画面回答，而不是让我上传图片。
- 作为用户，我希望回答能被读出来，因为手机场景下看屏幕不一定方便。

设计响应：

- 第一屏就是产品界面，不做 landing page。
- 相机预览占据主视觉区域。
- 语音识别、抓帧、AI 请求和 TTS 形成一个连续操作链。
- 底部移动触控栏固定高频动作，减少滚动寻找按钮。

### 用户 2：评审或演示者

场景：评审需要在较短时间内判断项目是否真实可运行，演示者需要在手机上完成闭环。

用户故事：

- 作为评审，我希望看到明确的依赖、运行方式和验证边界。
- 作为演示者，我希望权限失败时能知道失败在哪一步。
- 作为演示者，我希望没有语音输入时仍能演示视觉理解能力。

设计响应：

- README 明确列出依赖、环境变量、运行命令和验证方式。
- 摄像头、麦克风、抓帧、AI 请求各自有状态反馈。
- 增加截图提问兜底模式，点击画面即可使用默认视觉问题。

## 核心交互路径

```text
打开页面
  -> 授权摄像头
  -> 授权麦克风
  -> 语音提问
  -> 抓取当前帧
  -> 发送多模态请求
  -> 展示回答
  -> TTS 朗读 + 字幕
```

兜底路径：

```text
打开页面
  -> 授权摄像头
  -> 点击画面
  -> 抓取当前帧
  -> 使用默认视觉问题
  -> 展示回答
```

## 状态设计

### Orb 状态机

| 状态 | 触发条件 | 用户感知 |
| --- | --- | --- |
| `idle` | 未识别、未请求、未朗读 | 系统待命 |
| `listening` | Web Speech API 正在识别 | 系统正在听 |
| `thinking` | AI 请求进行中 | 系统正在理解画面 |
| `speaking` | TTS 正在朗读 | 系统正在回答 |

Orb 不承担点击入口，只承担状态可视化。交互入口放在按钮、预览画面和移动底部栏，避免状态实体变成含义不清的按钮。

### 对话状态层

相机开启后，Orb 旁边显示一个轻量状态层：

- 当前阶段：Idle / Listening / Thinking / Speaking
- 状态摘要：当前系统在做什么
- 链路信号：Question / Frame / Reply

这个状态层的设计原则是“可扫读”，不是说明书。它只显示用户判断下一步所需的状态，不解释功能。

## 失败恢复策略

| 失败类型 | 处理方式 | 用户影响 |
| --- | --- | --- |
| 摄像头拒绝或不可用 | 显示 Retry camera | 用户知道问题在相机授权或设备 |
| 麦克风拒绝或不可用 | 显示 Retry microphone | 用户知道问题在麦克风授权或设备 |
| 语音识别不可用 | 禁用识别入口并展示错误 | 避免用户点了没有反馈 |
| 抓帧失败 | 保留当前流程，提示重新抓帧 | 不丢失已识别问题 |
| AI 请求失败 | 展示错误并允许重试 | 不清空上下文和当前输入 |
| TTS 不可用 | 回答文本仍可读 | 保证核心结果不依赖音频 |

## 成本控制策略

### 不上传连续视频流

连续视频流会带来三个问题：

- 带宽高。
- 模型 token/图像处理成本不可控。
- 手机网络下延迟和稳定性差。

本项目只在用户提问或点击画面时抓取当前帧，默认 JPEG 压缩后作为 base64 data URL 发送。

用户影响：

- 响应更可控。
- 弱网下更容易完成演示。
- 费用与用户操作次数基本线性相关。

### 上下文只保留文本

多轮上下文只携带最近 4 轮文本问答，不重复上传历史图片。

原因：

- 历史图片通常成本高、价值低。
- 当前视觉问答应以当前帧为事实来源。
- 文本摘要足以保留追问意图。

用户影响：

- 可以自然追问。
- 不会因为多轮对话快速放大图像 token 成本。

### 浏览器端语音能力

语音识别和语音合成都使用浏览器 Web Speech API。

原因：

- 不需要服务端语音识别或 TTS 费用。
- 手机 Safari 场景下可用性足够支撑 MVP。
- 隐私边界更清晰，音频不进入项目后端。

限制：

- 浏览器兼容性和系统语音能力会影响体验。
- 语音识别需要用户手势和浏览器授权。

## 安全与隐私边界

- API key 不写入源码、`.env`、localStorage 或 sessionStorage，只保存在当前页面 React state。
- 不做登录，不绑定用户身份。
- 不做长期历史记录。
- 刷新页面后上下文清空。
- 摄像头和麦克风权限由浏览器控制，项目只在用户操作后请求。

## 取舍说明

### 为什么不做实时视频理解

实时视频理解更酷，但对 3 天 MVP 来说不是最直接路径。项目目标是可演示的视觉问答闭环，单帧抓取能覆盖主要场景，并且成本和稳定性更好。

### 为什么不做后端代理

最初计划阶段选择前端直连 API（API key 仅当前页面会话输入），MVP 早期阶段降低了部署复杂度。Day 3 已通过 PR20 完成前后端目录拆分与本地 Node 后端代理，API key 从浏览器完全移入 Node 进程 + `.env`，前端不再接触密钥。这是安全与演示可靠性的共同要求。

### 为什么保留右侧面板又增加底部触控栏

右侧面板适合桌面和完整状态查看；底部触控栏适合手机单手操作。两者复用同一套处理函数，不形成两套业务逻辑。

## 后续优化方向

以下按优先级排序，基于 2026-06-12 真机测试后的产品/UX 审计。

### P0：语音→抓帧→提问 自动化链路

**当前问题**：用户完成一次视觉问答需要五个独立操作（开启摄像头→开启麦克风→开始聆听→手动抓帧→手动提问），违背"能一步完成的不拆成三步"的设计原则。

**目标体验**：
- 用户说话→语音识别检测到完整问题→自动抓取当前帧→自动发送 AI 请求→自动朗读回答
- 用户不说话→点击画面→自动抓帧→默认视觉提问→回答
- "抓取画面"和"提问 AI"按钮保留为手动兜底，但不作为必经路径

**实现要点**：
- `useSpeechRecognition` 的 `speechend` / `finalTranscript` 事件触发自动抓帧
- 抓帧成功后自动调用 `askVisionQuestion`
- 端点检测超时从当前 1.5s 可调，避免过短截断问题

### P0：合并麦克风/语音识别为单一操作

**当前问题**：两个按钮（"开启麦克风" + "开始聆听"）暴露了浏览器权限模型的技术细节。用户心智是"我要对 AI 说话"，不是"我先开设备再开识别"。

**目标体验**：
- 一个按钮"开始对话"，按下后依次授权麦克风→自动开始语音识别
- 状态文案合并："正在听取你的问题"（隐含"麦克风已就绪"）
- 失败时区分"麦克风权限被拒绝"和"浏览器不支持语音识别"

### P1：Orb 居中 + 首次加载引导

**当前问题**：摄像头开启后 Orb 缩到 64px 置于右上角，手机上约指甲盖大小。"新颖性核心"失去视觉冲击力。首屏无明确行动号召。

**目标体验**：
- 摄像头未开启时：Orb 居中大幅展示，脉动呼吸动画，下方单一大按钮"开始对话"
- 摄像头开启后：AI 思考/说话时 Orb 在视频区中心短暂浮现（2-3s），表达处理状态后淡出
- 待机/聆听时 Orb 保持右上角但放大到 80-96px

### P1：AI 思考时的沉浸式等待体验

**当前问题**：AI 请求 3-6 秒延迟期间，只有右上角小 Orb 变金色+一行小字。用户不确定系统是否在工作。

**目标体验**：
- 视频区叠加半透明深色遮罩（opacity 0.4→0.6，渐入 300ms）
- Orb 瞬时放大到视频区中央，金色旋转动画
- 状态文案例："正在理解画面..." → "正在生成回答..."
- 回答到达后遮罩+Orb 淡出（300ms），视频恢复可见

### P2：精简右侧面板信息密度

**当前问题**：面板包含摄像头控件+语音控件+抓帧控件+AI 控件+上下文信息，手机上一屏看不完。多条状态文字重复当前系统状态。

**目标改进**：
- 面板默认只展示当前步骤和下一步操作，历史步骤折叠
- 去掉冗余 dot+文字状态条（与 DialogueStatus 组件信息重复）
- 上下文信息默认折叠，展开按钮显示"最近 N 轮对话"

### P3：回答后追问能力增强

**当前问题**：回答生成后，用户需要手动清空旧问题、重新开始聆听才能追问。口语化追问（"那这个怎么样"）需要上下文才能理解。

**目标改进**：
- 回答朗读完毕后自动进入聆听状态（不再需要手动点"开始聆听"）
- 连续对话模式下，麦克风保持开启，用户可以直接追问
- 静默超过 15 秒自动停止聆听，节省电量

### P3：弱网环境体验

**当前问题**：ngrok 隧道 + 手机蜂窝网络下，API 请求延迟可能超过 10 秒。无超时反馈。

**目标改进**：
- AI 请求超过 8 秒显示"网络较慢，正在等待 AI 响应..."
- 请求失败后保留已抓取的帧和已识别的问题，允许一键重试
- 大帧自动压缩更激进（quality 0.4→0.5），以 30-40KB 为目标

## 已完成扩展方向

- 增加可选低清/高清抓帧模式，让用户在速度和准确度之间选择。（待定：优先级 P3）
